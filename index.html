System Design

Architecture

The system is split across two compute platforms connected over CycloneDDS (ROS_DOMAIN_ID=19). The Raspberry Pi handles real-time hardware I/O; a separate PC runs perception and behaviour trees.

L1

Hardware Communication

OpenCR at /dev/ttyACM0 @ 57600 baud — custom binary protocol (magic 0x55AA, CRC8-CCITT) for 4 mecanum wheels + neck + 2 gripper servos. Arduino Nano at /dev/arduino_nano @ 115200 baud — 12-byte XOR-checksum packets for 4 corner LED strips. GPIO 27 drives RS-485 DE/RE direction pin.

L2

Sensor Processing & Odometry

mecanumbot_sensorproc_node converts OpenCR telemetry at 100 Hz: wheel ticks → mecanum forward kinematics → odometry + TF; IMU quaternion → sensor_msgs/Imu; battery voltage → BatteryState. TF (odom→base_footprint) feeds Nav2 AMCL.

L3

Perception (PC)

DR-SPAAM PyTorch model on the PC. Subscribes to /mecanumbot/scan, fills invalid points, resamples to 240 pts, runs inference, filters conf ≥ 0.85, transforms to map frame, publishes first detection as /mecanumbot/subject_pose.

L4

Navigation (Nav2)

Behaviour tree sends PoseStamped goals to /goal_pose. Nav2 does all path planning and obstacle avoidance, producing /cmd_vel which mecanumbot_io_node converts to wheel tick commands over serial.

L5

Behaviour Trees (PC)

Three py_trees configurations for different guiding experiments: bottom_up_tree (LED signals only), ctrl_tree (approach+turn, no signals), dog_tree (full dog-like with neck/gripper gestures + checkpoint following). All config via blackboard from behaviour_setting_constants.yaml.

MECANUM INVERSE KINEMATICS

scale = vel_tick/60 × 2π × wheel_radius
dist  = (sep_x + sep_y) / 2

BL = (Vx + Vy - Wz×dist) / scale   # back-left
BR = (Vx - Vy + Wz×dist) / scale   # back-right
FL = (Vx - Vy - Wz×dist) / scale   # front-left
FR = (Vx + Vy + Wz×dist) / scale   # front-right

Outgoing serial: struct.pack("<7h", BL, BR, FL, FR, N_pos, GL_pos, GR_pos)

MECANUM FORWARD KINEMATICS

dist  = (sep_x + sep_y) / 2

Vx = (BL + BR + FL + FR) / 4 × scale
Vy = (BL - BR - FL + FR) / 4 × scale
Wz = (-BL + BR - FL + FR) / 4 × scale / dist

Behaviour Tree Behaviours PC
mecanumbot_leading_behaviour · behaviours/
▼
py_trees behaviour classes that form the three guiding trees. All share a common blackboard populated by ConstantParamsToBlackboard at startup.
ClassTypeWhat it does
ConstantParamsToBlackboardActionReads YAML config, writes all mission params to blackboard. Runs once on startup.
DistanceToBlackboardActionComputes euclidean distances: robot↔subject, subject↔target, robot↔target. Updates blackboard every tick.

DogBehaviourSequenceActionSends timed AccessMotorCmd sequences for neck nods and gripper opens — the "dog" gesture animations.
DogCheckFollowingConditionCompares subject distance from last tick to current. If delta > Dog_following_max_threshold the subject is "lost".
DogSelectTargetActionFinds closest checkpoint to robot, then increments index by 1. Robot always drives to the next checkpoint, not the nearest.

Watch Out

15 Gotchas

Non-obvious things that will surprise you. Read before touching any code.

01
Packet Format Mismatch Between YAML and Code Critical
mecanumbot_IO_node.py vs mecanumbot.yaml
Code default for payload_fmt is <27h14f (27 × int16 + 14 × float32 = 82 bytes) but YAML overrides to <23h14f (74 bytes). The four err_bl/br/fl/fr error fields are never populated when running from YAML. Similarly, baudrate defaults to 1 000 000 in code but YAML sets it to 57 600. The YAML always wins. If you update the OpenCR firmware, check both.
02
100 ms Serial Write Delay Limits Command Rate to 10 Hz Warning
mecanumbot_IO_node.py · update_motor_cmds_out()
A time.sleep(0.1) is held while cmd_lock is acquired. GPIO 27 is toggled HIGH before the write and LOW after the sleep — RS-485 direction control. Motor commands are capped at ~10 Hz even if /cmd_vel fires faster.
03
All Three Behaviour Trees Share the Same Node Name Critical
bottom_up_tree.py, ctrl_tree.py, dog_tree.py
All three call tree_node.setup(node_name="bottom_up_tree_node"). Running two trees on the same machine causes a node name collision. Both ctrl_tree.py and bottom_up_tree.py executables register as bottom_up_tree_node.
04
accessory_monitor Calls main() Recursively Bug
mecanumbot_monitor/accessory_monitor.py ~line 79
Inside the finally block: main() is called again after shutdown. Every keyboard interrupt or error grows the stack. wheel_monitor.py does NOT have this bug.
05
DR-SPAAM Always Uses the First Detection as Subject Warning
mecanumbot_lidar_detect_people.py · _parse_subject_pose()
DR-SPAAM returns detections in scan order, not sorted by confidence or proximity. In a multi-person environment, poses[0] can jump between different people as they move. There is no person tracking or identity assignment.
06
DR-SPAAM XY Axes Are Swapped and Negated Info
mecanumbot_lidar_detect_people.py
dets_xy = -1 * dets_xy then p.position.x = xy[1]; p.position.y = xy[0]. Both operations adjust for detector internal convention vs. ROS convention (x forward, y left). If you change LiDAR mount orientation or swap detectors, revisit this.
07
read_feedback() Has Global Singleton State Info
mecanumbot_led/led_control_node.py
State machine variables are stored on the function object itself (read_feedback.__dict__). Multiple LedServiceNode instances would share the same state machine. Fine for single-node use.
08
LED duration_ms Is Hardcoded at 1000 ms Info
mecanumbot_led/led_control_node.py
self.duration_ms = 1000 is never changed. The SetLedStatus service has no duration field. The Arduino always receives a 1 second duration. Must edit Python source to change it.
09
DogSelectTarget Always Targets the Next Checkpoint, Not the Closest Warning
mecanumbot_leading_behaviour/behaviours/dog_behaviours.py
After finding the closest checkpoint to the robot, closest_index += 1. The robot always drives to the checkpoint after the nearest one. Ensures forward progress but can cause large jumps if the robot is placed out of sequence.
10
Velocity Clamping Is Silent Info
mecanumbot_IO_node.py · vel_cmd_callback()
Wheel velocities are clamped to ±300 ticks with no warning logged. Teleop max (0.234 m/s) produces exactly ~300 ticks so the clamp never fires in normal operation. A behaviour tree sending large velocities will be silently saturated.
11
Scan Is Resampled to 240 Points — Hardcoded Warning
mecanumbot_lidar_detect_people.py
240 is a hardcoded default argument in scan_callback. The LD08 produces a different number of points, so interpolation always runs. If you swap to a LiDAR with a different count, interpolation still runs. If the DR-SPAAM model was trained with a different count, inference quality will degrade.
12
Wheel Joint Positions Are Always Zero Info
mecanumbot_sensorproc_node.py · set_joint_state()
The four wheel joint positions are hardcoded to 0.0. The OpenCR reports pos_bl/br/fl/fr but they are not used here. Wheel meshes in RViz do not rotate.
13
SensorState.msg Is Orphaned TurtleBot3 Baggage Info
mecanumbot_msgs/msg/SensorState.msg
Copied directly from original turtlebot3_msgs. No node publishes or subscribes to it. Safe to ignore; do not delete unless you are sure no external tools depend on it.
14
Behaviour Tree Tick Rates Differ Significantly Info
bottom_up_tree.py (100 Hz), dog_tree.py (10 Hz), ctrl_tree.py (100 Hz)
The dog tree ticks at 10 Hz — 10× slower than the others. Intentional to avoid flooding Nav2 with goals. But state changes (subject lost, goal succeeded) are only evaluated every 100 ms in dog mode.
15
mecanumbot_core_parameters.yaml File Was Not Found Warning
launch_mecanumbot_base.launch.py
The launch file references mecanumbot_description/param/mecanumbot_core_parameters.yaml but this file was not found at that path. The actual config lives in mecanumbot.yaml in the same directory. Either generated at build time or the launch file has a path mismatch. Verify before deploying.
