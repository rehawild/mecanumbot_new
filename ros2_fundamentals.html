Concept 01

What is ROS2?

A communication framework that lets software modules on one or many computers talk to each other as if they were in the same room.

A robot is not one program — it's dozens of programs running simultaneously: a motor driver, a sensor reader, a path planner, a camera processor, a behaviour controller. Each of these needs to send and receive data from the others, often across different hardware, programming languages, and even different physical machines.

ROS2 solves this by giving every piece of software a common language for announcing what it has (publications) and asking for what it needs (subscriptions, service calls). Under the hood it uses a mature industrial protocol called DDS that handles discovery, serialization, quality-of-service, and networking automatically. You just write the logic; ROS2 handles the plumbing.

ROS2 is the successor to ROS1. The main improvements: no more single master process (if one node crashes the rest keep running), real-time support, better security, and proper multi-machine networking out of the box.

The ROS2 Graph — Abstract View

In Mecanumbot src/mecanumbot/ · src/mecanumbot_remote/ · src/mecanumbot_behaviours/

The robot splits its workload across two physical computers: the Raspberry Pi on the robot and an operator's PC nearby. Both run ROS2 nodes. Because they share the same DDS domain (ROS_DOMAIN_ID=19) and are on the same WiFi network, every topic and service is accessible from either machine — the behaviour tree on the PC can read /mecanumbot/scan produced by the LiDAR on the Pi as if it were local.

The Pi runs: mecanumbot_io_node, mecanumbot_sensorproc_node, mecanumbot_led_service_node, the LD08 LiDAR driver, and robot_state_publisher.

The PC runs: DrSpaamNode (person detection), the behaviour tree nodes (doglike_leading_bt_node etc.), teleoperation nodes, and Nav2.

Quick Facts

  • ROS stands for Robot Operating System — it's a middleware, not a real OS.
  • Version: This project uses ROS2 Humble on Ubuntu 22.04.
  • Language: Python and C++ are both first-class. Mecanumbot uses pure Python.
  • No master process: In ROS1 a central roscore had to run. In ROS2 nodes discover each other via DDS. Kill any node and the rest keep going.
  • Four communication patterns: Topics (async, broadcast), Services (sync, 1-to-1), Actions (long-running with feedback), Parameters (node configuration).

Concept 02

Nodes

A node is a single process that does exactly one job in the robot system.

Think of a node like a Unix process with superpowers. It starts up, it has a name, it registers itself with the ROS2 network, and then it either runs in a loop (spinning, processing messages) or sleeps waiting for input. The rule of thumb: one node, one responsibility. The motor driver is one node. The sensor converter is another. The person detector is a third.

This separation has a practical payoff: you can kill and restart any one node without touching the others. You can run nodes on different machines. You can swap in a fake "mock" node for testing. If nodes were all tangled in one giant program, none of this would be possible.

In Python, a ROS2 node is a class that inherits from rclpy.node.Node. The constructor registers the node name. Then a spin() call keeps it alive and processing callbacks.

Node Anatomy

In Mecanumbot src/mecanumbot/mecanumbot_core/mecanumbot_core/mecanumbot_IO_node.py

Every Python node in this codebase follows the same skeleton:

class MecanumbotIONode(Node): — inherits from ROS2 Node
super().__init__('mecanumbot_io_node') — registers the name on the network
self.create_subscription(...) — hooks up incoming topic handlers
self.create_publisher(...) — creates an outgoing stream
self.create_timer(0.01, self.timer_cb) — runs a function every 10 ms
rclpy.spin(node) in main() — keeps the node alive

The nine nodes in this repo are: mecanumbot_io_node, mecanumbot_sensorproc_node, mecanumbot_led_service_node, accessory_monitor, wheel_monitor, DrSpaamNode, MecanumbotJoyTeleop, mecanumbot_ledgui_client, plus the three behaviour tree runner nodes.

Quick Facts

  • Node names must be unique within a domain. Two nodes with the same name cause silent conflicts — see Gotcha #3 in the project docs.
  • Namespaces: A node can be placed under a namespace like /mecanumbot/. All its topics are automatically prefixed. This is why most topics in this project start with /mecanumbot/.
  • Lifecycle nodes: ROS2 has a more advanced "managed" node with explicit configure/activate/deactivate states. Mecanumbot uses simple (unmanaged) nodes throughout.
  • Spinning: rclpy.spin(node) blocks forever and processes callbacks. spin_once() processes one pending callback. The behaviour trees use a custom executor instead of plain spin().

Concept 03

Topics & Pub/Sub

A topic is a named data stream — any node can put data in (publish), any node can read data out (subscribe), and they never need to know about each other.

Topics are the primary way ROS2 nodes share data. A topic has a name (e.g. /mecanumbot/scan) and a type (e.g. sensor_msgs/LaserScan). The publisher sends messages whenever it has new data. The subscriber receives them asynchronously via a callback function. Neither side needs to know the other exists.

This decoupling is powerful: you can add a new subscriber (e.g. a data recorder) without touching the publisher. You can run the publisher on the Pi and the subscriber on the PC with no code changes. You can kill the subscriber and the publisher keeps running.

Topics are best for continuous data streams: sensor readings, odometry, velocity commands. They're not the right choice when you need a guaranteed response — use Services for that.

One Publisher, Multiple Subscribers

In Mecanumbot The three most important topics to understand

/mecanumbot/opencr_state — The heartbeat of the hardware. io_node publishes this at 100 Hz with all wheel velocities, IMU data, battery voltage, and servo positions. sensorproc_node subscribes to it and converts it into standard ROS2 messages. If this topic stops flowing, the entire sensor pipeline dies silently.

/cmd_vel — The "move here" command. Nav2 or the teleop joystick publishes a Twist message here. io_node subscribes and converts it to four wheel velocity ticks via mecanum inverse kinematics. This is the single control interface for the wheels.

/mecanumbot/scan — The LiDAR point cloud. Produced by the LD08 driver, consumed by both Nav2 (obstacle avoidance) and DrSpaamNode (person detection). One publisher, two subscribers — neither knows about the other.

Quick Facts

  • QoS: Topics have a Quality of Service policy — reliability (best-effort vs guaranteed), durability, history. Mismatched QoS between publisher and subscriber means messages are silently dropped.
  • Remapping: Topic names can be remapped at launch. io_node internally subscribes to /cmd_vel but this gets remapped in the launch file to /mecanumbot/cmd_vel.
  • No history by default: If a subscriber starts after the publisher, it misses all previous messages. Use transient_local durability if you need late-joining subscribers to get the last message.
  • Bandwidth: At 100 Hz, opencr_state is a significant stream. Over WiFi between Pi and PC this can cause latency if the network is congested.

Concept 04

Services

A service is a synchronous call-and-response — the caller asks, the server does something, and the caller waits for a result.

Topics are great for continuous streams of data, but sometimes you need a guarantee: "do this thing and tell me if it worked." That's a service. The client sends a Request, the server processes it and sends back a Response. The client blocks (or uses an async callback) until it gets the response.

A service has a name (like /set_led_status) and a type (like mecanumbot_msgs/srv/SetLedStatus) that defines the shape of the request and response. Only one server should exist per service name. Many clients can call it.

Use a Topic when…

  • Data flows continuously (sensor readings, velocity)
  • You don't need a response
  • Multiple subscribers need the same data
  • Losing an occasional message is acceptable

Use a Service when…

  • You need a success/failure confirmation
  • It's a one-off request (turn on LED, query state)
  • The result must be acted on before proceeding
  • Only one thing should handle the request
In Mecanumbot src/mecanumbot/mecanumbot_led/mecanumbot_led/led_control_node.py

The LED control uses services because the behaviour tree needs to know the LEDs were actually set before continuing. When LEDBehaviourSequence calls /set_led_status, it passes a mode and colour for all four corner LED strips, then waits for a bool success response before the behaviour returns SUCCESS or FAILURE.

The LedServiceNode is the server — it receives the request, builds a 12-byte packet (with XOR checksum), sends it over serial to the Arduino Nano, reads the Arduino's acknowledgment, and returns the result.

The mecanumbot_ledgui_client (the Tkinter GUI) is a separate client of the same service — you can use both simultaneously without changing any server code.

ServiceTypeServerWhat it does
/set_led_statusSetLedStatusled_service_nodeSet animation mode + colour for all 4 LED strips. Returns bool success.
/get_led_statusGetLedStatusled_service_nodeRead back the current LED state. Used by the GUI.

Quick Facts

  • Synchronous by default: The client blocks until the server responds. If the server crashes mid-call, the client hangs. Always set a timeout when calling services in production.
  • Service types: Defined in .srv files with a --- separator between request fields (above) and response fields (below).
  • Not for streaming: Calling a service in a tight loop is inefficient and defeats the purpose. Use a topic instead.
  • Only one server: Unlike topics (where many publishers can exist), a service should have exactly one server. Two servers on the same service name cause undefined behavior.

Concept 05

Actions

An action is like a service that can take a long time — with progress feedback and the ability to cancel mid-flight.

Some robot tasks take seconds or even minutes: "navigate to this pose", "pick up the object", "scan the room". A plain service can't handle these — the caller would block indefinitely. An action solves this with three-phase communication: Goal (client sends what to do), Feedback (server sends progress updates), Result (server sends final outcome).

Crucially, the client can cancel a goal at any time. Nav2 uses actions heavily — "navigate to pose" is an action where the robot sends periodic feedback about its current position and estimated time to goal.

Action Flow Timeline

In Mecanumbot src/mecanumbot_behaviours/mecanumbot_leading_behaviour/behaviours/movement_managers.py

The behaviour tree uses the Nav2 /navigate_to_pose action to move the robot. The TurnToward and Approach behaviour classes send a PoseStamped goal, then monitor the action status on the /navigate_to_pose/_action/status topic. They return RUNNING while Nav2 is still executing, and SUCCESS or FAILURE when the action completes.

This is why the behaviour tree doesn't block — it just polls the action status each tick (every 10 ms or 100 ms depending on the tree). The robot is physically moving during all those RUNNING ticks.

Note: the behaviours send goals via the simpler /goal_pose topic rather than the full action client API, and read the action status separately. This is a lightweight alternative to using an ActionClient directly.

Quick Facts

  • Three components: Goal (client → server), Feedback (server → client, repeated), Result (server → client, once at end).
  • Cancellable: Client can cancel a goal at any time. Server should respect cancellation requests gracefully.
  • Status values: ACCEPTED, EXECUTING, CANCELING, SUCCEEDED, CANCELED, ABORTED.
  • Under the hood: Actions are built on top of topics and services. ROS2 creates a bundle of communication channels automatically when you define an action server.
  • Nav2 actions in this project: /navigate_to_pose, /follow_path, /compute_path_to_pose.

Concept 06

Messages

A message is a typed data structure — it defines exactly what information flows through a topic or a service.

Every topic has a type. /cmd_vel carries geometry_msgs/Twist. /mecanumbot/scan carries sensor_msgs/LaserScan. The type is defined in a .msg file that lists fields with their types, like a struct or dataclass. ROS2 compiles these definitions into code so your Python gets auto-complete and type safety.

ROS2 ships with a large library of standard messages: std_msgs, geometry_msgs, sensor_msgs, nav_msgs. When the standard messages don't fit, you define custom ones — which is exactly what this project does in mecanumbot_msgs.

In Mecanumbot src/mecanumbot_msgs/mecanumbot_msgs/msg/

OpenCRState.msg — The biggest custom message. It carries the entire telemetry snapshot from the OpenCR firmware: 4 × commanded velocity, 4 × measured velocity, 4 × position, 4 × current, 4 × acceleration (all as int16 ticks), plus 3 servo positions, battery voltage, and 13 IMU floats. Total: 27 int16 fields + 14 float32 fields. The YAML overrides payload format to <23h14f — so the four err_* fields exist in the message definition but are always zero.

AccessMotorCmd.msg — Three float32 fields: n_pos (neck), gl_pos (left gripper), gr_pos (right gripper). Values in radians, scaled ×100 inside io_node before transmitting over serial.

Twist (from geometry_msgs) — Used for /cmd_vel. Two vectors: linear (x, y, z velocity in m/s) and angular (x, y, z angular velocity in rad/s). For a ground robot, only linear.x (forward), linear.y (strafe), and angular.z (rotation) matter. Mecanum wheels use all three.

Quick Facts

  • Field types: bool, int8/16/32/64, float32/64, string, time, duration, and arrays of any of these. Can nest other message types.
  • The header field: Most sensor messages include a std_msgs/Header with a timestamp and frame_id. The frame_id tells you which coordinate frame the data is in — essential for TF.
  • .msg files: Each line is type field_name. Lines starting with # are comments. A --- separator divides request/response in .srv files.
  • Serialization: Messages are automatically serialized/deserialized by DDS. You never touch raw bytes in Python.

Concept 07

Parameters & YAML

Parameters are named configuration values attached to a node — they let you change behaviour without touching the code.

Hardcoding values like baud rates, wheel radii, or confidence thresholds inside source code is fragile — every change requires a recompile or at least a code edit. ROS2 parameters solve this: a node declares what values it accepts (with types and defaults), and at launch time a YAML file can override any of them. The node reads the values with self.declare_parameter() and self.get_parameter().value.

Parameters can also be changed at runtime without restarting the node, via ros2 param set. Nodes can register a callback to respond to changes immediately.

In Mecanumbot src/mecanumbot/mecanumbot_description/param/mecanumbot.yaml

The mecanumbot.yaml file is loaded at launch by launch_mecanumbot_base.launch.py. It overrides the default values declared in the node constructors. Two overrides that matter:

Baud rate: mecanumbot_IO_node.py declares dev_params.baudrate with default 1000000. The YAML sets it to 57600. Without this YAML the serial port would open at the wrong speed and receive only garbage.

Packet format: The code default is <27h14f (82 bytes) but the YAML sets <23h14f (74 bytes). If you update the OpenCR firmware and it starts sending 82-byte packets, you must change this YAML — otherwise every packet will be misread.

The YAML is organized as nested keys: node_name.ros__parameters.group.field: value. The ros__parameters key (double underscore) is the ROS2 convention.

# mecanumbot.yaml structure (simplified)
mecanumbot_io_node:
  ros__parameters:
    dev_params:
      device_name: "/dev/ttyACM0"
      baudrate: 57600          # overrides code default of 1000000
    packet_params:
      payload_fmt: "<23h14f"  # overrides code default of <27h14f
    robot_params:
      wheel:
        radius: 0.0325
        separation_x: 0.129
        separation_y: 0.300
⚠ Code default vs YAML override: When reading a node's source code, the value in declare_parameter('x', DEFAULT) is not what the node uses in production. Always check the YAML file too. In mecanumbot, the baud rate and packet format defaults in the code are both wrong for the actual hardware — only the YAML makes it work.

Quick Facts

  • Declare first: A node must call declare_parameter(name, default) before it can use get_parameter(name). Trying to get an undeclared parameter throws an exception.
  • Types: bool, int, float, string, lists of these. Type is inferred from the default value or can be declared explicitly.
  • Nested params: Accessed with dot notation: dev_params.baudrate. The YAML mirrors this with nested keys.
  • Runtime changes: ros2 param set /node_name param_name value changes a parameter live. Not all nodes respond to changes — depends on whether a callback was registered.

Concept 08

Launch Files

A launch file is a Python script that starts multiple nodes at once, with their parameters, namespaces, and remappings configured in one place.

A robot has many nodes. Starting them one by one with ros2 run is tedious and error-prone — you'd have to get the order, parameters, and namespaces exactly right every time. A launch file automates this. It's a Python script that returns a LaunchDescription — a list of things to start.

Launch files can: start nodes (with parameters), include other launch files, set environment variables, remap topic names, conditionally include things based on arguments. They're composable — the mecanumbot base launch includes a separate state publisher launch, which includes the robot description.

In Mecanumbot src/mecanumbot/mecanumbot_bringup/launch/launch_mecanumbot_base.launch.py

The base launch file starts everything the robot needs to move and sense. Key things it does:

  1. Locates the parameter file — uses get_package_share_directory to find mecanumbot.yaml regardless of install path.
  2. Starts mecanumbot_io_node — loads YAML params, sets namespace to /mecanumbot, remaps /cmd_vel to /mecanumbot/cmd_vel.
  3. Starts mecanumbot_sensorproc_node — same YAML, same namespace.
  4. Starts the LD08 LiDAR driver — device /dev/ttyUSB0, output topic /mecanumbot/scan.
  5. Includes the state publisher launch — which starts robot_state_publisher with the URDF model.

The namespace='mecanumbot' argument in the node declaration is why all topics from these nodes appear under /mecanumbot/. Without it, opencr_state would be at /opencr_state and would collide with any other robot on the network.

# Skeleton of a launch file
from launch import LaunchDescription
from launch_ros.actions import Node
from ament_index_python.packages import get_package_share_directory

def generate_launch_description():
    params = get_package_share_directory('mecanumbot_description') + '/param/mecanumbot.yaml'

    return LaunchDescription([
        Node(
            package='mecanumbot_core',
            executable='mecanumbot_io_node',
            name='mecanumbot_io_node',
            namespace='mecanumbot',
            parameters=[params],
            remappings=[('/cmd_vel', '/mecanumbot/cmd_vel')],
        ),
        # ... more nodes
    ])

Quick Facts

  • Must define generate_launch_description(): This is the entry point ROS2 calls. It must return a LaunchDescription.
  • IncludeLaunchDescription: You can nest launch files. The base launch includes the state publisher launch, keeping concerns separate.
  • Arguments: Launch files can accept command-line arguments with DeclareLaunchArgument, letting you e.g. toggle simulation mode.
  • Package discovery: get_package_share_directory('pkg_name') finds the installed share directory — the right way to locate YAML/URDF files at launch time.

Concept 09

TF & Coordinate Frames

TF maintains a live tree of coordinate frames and can instantly answer "where is point X in frame Y right now?"

A robot's sensors are attached at different places and orientations. The LiDAR is at the robot's centre. The camera is on the neck servo which is angled. The wheel encoders tell you how far the robot moved in the world. When the DR-SPAAM detector says "person at (1.2, 0.4) relative to the LiDAR", you need to convert that to map coordinates before giving it to the behaviour tree. That conversion requires knowing the full chain of transforms from LiDAR to robot body to odom to map — at the exact right timestamp.

TF does this automatically. Nodes broadcast transforms (parent frame → child frame, at a given time). TF accumulates them into a tree. Any other node can then look up any transform in the tree, even between frames that don't share an edge, by chaining through the tree.

Mecanumbot TF Tree

In Mecanumbot mecanumbot_sensorproc_node.py · robot_state_publisher · Nav2 AMCL

Who broadcasts what:

mecanumbot_sensorproc_node broadcasts odom → base_footprint at 100 Hz. This is the wheel-odometry transform — it integrates the mecanum forward kinematics to give the robot's estimated position relative to where it started.

robot_state_publisher reads the URDF model and broadcasts all the fixed transforms: base_footprint → base_link → imu_link → laser_link → camera_link, etc. These never move relative to each other (unless the neck servo moves, which updates the neck joint state).

Nav2's AMCL node broadcasts map → odom. This is the localization correction — it shifts the odom frame so the robot's position in the map frame matches the particle filter estimate. Without this transform, the robot has no idea where it is on the pre-built map.

DR-SPAAM transforms its detections from laser_link frame to map frame using TF lookups — that's how a LiDAR reading becomes a map-frame position the behaviour tree can navigate to.

Key insight: When you see frame_id in a message header, that tells you which frame the data is expressed in. sensor_msgs/LaserScan from the LD08 has frame_id: "laser_link". To use those points in path planning you need to transform them to the map frame. TF makes that a single function call.

Quick Facts

  • TF tree must be connected: Every frame must have a path to every other frame via the tree. A disconnected frame causes tf2.LookupException. If a node dies, its transforms stop updating and lookups start throwing ExtrapolationException.
  • Static vs dynamic transforms: Static transforms (fixed mounting positions from URDF) are broadcast by robot_state_publisher. Dynamic transforms (odometry, localization) are broadcast by sensor processing nodes.
  • Time matters: Every transform is stamped. TF interpolates between the nearest timestamps. If you ask for a transform at a time too far in the past, it fails.
  • base_footprint vs base_link: base_footprint is the robot's position projected onto the floor (z=0). base_link is the robot's physical centre, slightly above the floor. Nav2 prefers base_footprint for 2D navigation.

Concept 10

Packages & Workspaces

A package is the unit of code organisation in ROS2; a workspace is the folder that contains all the packages you're building together.

All ROS2 code lives in packages. A package has a package.xml (metadata: name, version, dependencies) and either a CMakeLists.txt (C++ or mixed) or a setup.py (pure Python). The build tool colcon scans the workspace for packages, figures out the dependency order, and builds them all.

After building, nodes are run with ros2 run package_name executable_name. Launch files are run with ros2 launch package_name file.launch.py. The install/ directory (created by colcon) is what ROS2 actually uses at runtime — not src/.

In Mecanumbot src/ — four top-level source groups

The workspace root is D:/mecanumbot_new/. Source code lives in src/. After running colcon build, the install/ directory is created with all compiled packages.

The meta-package pattern: mecanumbot and mecanumbot_behaviours are "meta-packages" — they contain only a package.xml with no code. Their purpose is grouping: sudo apt install ros-humble-mecanumbot would install all sub-packages in one command (if this were a released package).

Quick Facts

  • Source this after building: source install/setup.bash makes all built packages visible to ros2 commands in your terminal. Without sourcing, ros2 run mecanumbot_core ... says "package not found".
  • --symlink-install: colcon build --symlink-install makes Python nodes editable in place — changes to .py files take effect without rebuilding. Recommended for development.
  • Dependencies: package.xml lists depend, build_depend, and exec_depend tags. rosdep install --from-paths src installs all system dependencies listed there.
  • Executables: Python nodes are registered as entry points in setup.py under console_scripts. That's what maps mecanumbot_io_node (the executable name) to the Python function main().

Concept 11

DDS & Networking

DDS (Data Distribution Service) is the transport layer under ROS2 — it handles node discovery, message serialization, and network communication automatically.

In ROS1, a central roscore process managed the network. Everything depended on it — if it crashed, everything crashed. ROS2 replaced this with DDS, an industrial middleware standard. DDS is completely decentralised: nodes find each other via multicast UDP. When a new node starts, it announces itself and DDS discovers all peers on the same network that share the same Domain ID.

There are multiple DDS implementations (vendors). ROS2 supports all of them, but they're not always fully interoperable. This project specifies CycloneDDS — it's known for good performance over WiFi and between machines, which is exactly what this project needs (Pi ↔ PC communication).

Pi ↔ PC over CycloneDDS

In Mecanumbot Environment variables — set before any ros2 command

Two environment variables must be set consistently on both the Pi and the PC:

ROS_DOMAIN_ID=19 — This is like a network channel. All nodes with the same Domain ID see each other's topics and services. Nodes with a different Domain ID are completely invisible to each other. Domain 19 is arbitrary — it just needs to match on both machines. If your neighbour's robot also uses Domain 19, your topics will collide.

RMW_IMPLEMENTATION=rmw_cyclonedds_cpp — This tells ROS2 to use CycloneDDS as the DDS vendor. Both machines must use the same DDS implementation to communicate. CycloneDDS performs better than the default FastDDS over WiFi and handles the Pi ↔ PC link more reliably.

These are usually set in ~/.bashrc so they apply to every terminal automatically.

⚠ Same network, same domain = same graph: Any machine on the same WiFi with ROS_DOMAIN_ID=19 and CycloneDDS will see and interact with your robot. This means another developer on your network could accidentally send /cmd_vel to your robot. In a lab setting, consider using different Domain IDs per robot to isolate them.

Quick Facts

  • Domain IDs: Range 0–101. Avoid 0 (the default — anyone with a default install would join). Pick something non-obvious like 19, 42, 77.
  • DDS vendors: FastDDS (default), CycloneDDS, Connext, Gurum. This project requires CycloneDDS. sudo apt install ros-humble-rmw-cyclonedds-cpp.
  • Multicast discovery: DDS uses UDP multicast to discover peers. Some enterprise WiFi networks block multicast, breaking cross-machine discovery. In that case, configure CycloneDDS with a unicast peer list.
  • QoS profiles: DDS also manages message reliability, history depth, and durability. Mismatched QoS between publisher and subscriber causes silent message dropping — one of the harder bugs to diagnose.

Useful Terminal Commands

# Set up environment (put these in ~/.bashrc)
export ROS_DOMAIN_ID=19
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp

# Verify DDS implementation in use
ros2 doctor --report | grep DDS

# Check if you can see nodes from the other machine
ros2 node list
# Should show nodes from BOTH Pi and PC if connected

# Diagnose network/DDS issues
ros2 doctor
Reference

Glossary

Quick definitions for terms you'll encounter reading the codebase.

Node
A single ROS2 process with one responsibility. Inherits from rclpy.node.Node. Kept alive by rclpy.spin().
Topic
A named, typed, asynchronous data stream. Any node can publish; any node can subscribe. No coupling between them.
Service
Synchronous call-and-response. Client sends Request, server processes it and returns Response. Caller blocks until response arrives.
Action
Like a service but for long-running tasks. Sends a Goal, receives periodic Feedback, gets a final Result. Cancellable at any time.
Message (.msg)
A typed data structure for topic communication. Defined in a .msg file. Compiled into Python/C++ classes by colcon.
Parameter
A named configuration value attached to a node. Declared in code with a default, overridden at launch time via YAML.
Launch file
A Python script that starts multiple nodes with their parameters, namespaces, and topic remappings configured in one place.
TF / tf2
Transform library. Maintains a tree of named coordinate frames and answers "where is X in frame Y at time T?" for any two connected frames.
DDS
Data Distribution Service — the transport layer under ROS2. Handles node discovery, serialization, and networking. Replaces ROS1's roscore.
ROS_DOMAIN_ID
A number (0–101) that acts as a network channel. Only nodes with the same Domain ID see each other. Set to 19 in this project.
Namespace
A prefix applied to a node's topics. namespace='mecanumbot' makes /opencr_state become /mecanumbot/opencr_state.
Remapping
Renaming a topic/service at launch time. ('/cmd_vel', '/mecanumbot/cmd_vel') reroutes commands without changing node code.
Spin
Calling rclpy.spin(node) blocks the main thread and continuously processes incoming messages and timer callbacks.
QoS
Quality of Service. Controls reliability (best-effort vs reliable), durability (volatile vs transient_local), and history depth. Must match between publisher and subscriber.
colcon
The ROS2 build tool. Scans src/, resolves dependencies, builds packages in order, outputs to install/.
ament
The ROS2 build system (underlying colcon). ament_python for pure Python packages, ament_cmake for C++ or mixed packages.
CycloneDDS
The DDS vendor used in this project. Better WiFi performance and multi-machine reliability than the default FastDDS. Set via RMW_IMPLEMENTATION.
Nav2
ROS2 Navigation2 stack. Handles AMCL localisation, costmap generation, path planning, and velocity control. Accepts goals on /goal_pose.
AMCL
Adaptive Monte Carlo Localisation. Nav2's particle-filter localiser. Reads /scan + /odom, publishes the map→odom TF transform.
py_trees
Python behaviour tree library. Nodes return SUCCESS, FAILURE, or RUNNING each tick. Used in this project for the robot guiding logic.
Blackboard
py_trees shared key-value store. Behaviour nodes read/write config and state here instead of passing data directly between each other.
frame_id
The header.frame_id field in stamped messages tells you which coordinate frame the data is expressed in. Essential for TF lookups.
rosdep
Dependency installer. rosdep install --from-paths src reads package.xml dependencies and installs them via apt/pip.
Executor
Runs the event loop for one or more nodes. rclpy.spin() uses a single-threaded executor. MultiThreadedExecutor processes callbacks in parallel.
